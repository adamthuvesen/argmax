Plan: Excellent HardBench Docs

01 — Summary

Build docs for two audiences: benchmark users who want to run and audit HardBench, and contributors who want to author or maintain tasks. Keep README.md as the fast path, then move deeper material into focused docs/ pages.

Success means a new reader can install HardBench, run smoke and frontier benchmarks, understand artifacts and scores, add a custom agent, and author a valid benchmark task — without reading the source first.

02 — Key Changes
  • A. Restructure README.md into a crisp overview
    • what HardBench is, in three sentences
    • quick start: install, run smoke, read result
    • common run commands and flags
    • result and artifact interpretation
    • links into the deeper docs
  • B. Add docs/running-benchmarks.md for operator workflows
    • CLI reference for list, run, grade, report, and export eval-audit
    • concurrency, timeouts, run IDs, output directories, and cost caveats
    • how to inspect runs/<run-id>/, summary.md, results.jsonl, diffs, transcripts, and grader output
  • C. Add docs/agents.md for custom agent configuration
    • built-in agent expectations and defaults
    • full YAML schema for agent files
    • stdin vs {prompt} command passing
    • required env vars and local authentication assumptions
  • D. Add docs/task-authoring.md for benchmark contributors
    • required task layout: task.yaml, workspace/, hidden/, reference/
    • task.yaml fields and defaults inherited from TaskSpec
    • preserved public return shapes, visible tests, hidden grader rubric
