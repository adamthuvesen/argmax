## A. Dismissal consistency (Escape / outside-click / focus-restore)

- [x] **A1 (H)** [KeyboardCheatSheet.tsx:13-29](src/renderer/components/KeyboardCheatSheet.tsx:13) — Esc handler is on the overlay div (`onKeyDown`), not global. The overlay has `tabIndex={-1}` but nothing focuses it on open, and the typing-target guard in [useOverlays.ts:80](src/renderer/hooks/useOverlays.ts:80) blocks Esc while focus is in the composer textarea. Result: opening the cheat sheet via Cmd+/ while focus is in the composer means Esc cannot close it. **Repro:** focus composer → Cmd+/ → Esc → nothing happens. **Expected:** Esc closes regardless of focus location. **Fix:** rely on `useOverlays` Esc precedence (already includes cheat sheet) and ensure it does *not* skip on typing-target when an overlay is open and obscures the textarea — or focus the cheat-sheet dialog on open. *Status:* commit:8c4d97c (adopted useDismissOnOutsideOrEscape).
- [x] **A2 (M)** [KeyboardCheatSheet.tsx](src/renderer/components/KeyboardCheatSheet.tsx) — No focus restore on close. `previousActiveElement` not captured. **Fix:** capture on open, restore on close, matching CommandPalette/SearchOverlay/CommitDialog. *Status:* commit:8c4d97c.
- [x] **A3 (M)** [CommandPalette.tsx:298-306](src/renderer/components/CommandPalette.tsx:298) — Esc is handled only on the input via `onKeyDown`; if focus drifts to a result row (mouse hover doesn't move focus, so this is theoretical) Esc would miss. The outside-click handler is on overlay `onMouseDown` (works). **Fix:** lift Esc to a document keydown OR adopt `useDismissOnOutsideOrEscape` against the inner palette ref. *Status:* commit:351436f.
- [x] **A4 (M)** [CommandPalette.tsx](src/renderer/components/CommandPalette.tsx) — Does not scroll the currently selected result into view when ArrowUp/Down moves past the viewport. Long result lists hide the active row. **Fix:** `scrollIntoView({ block: "nearest" })` on selectedIndex change. *Status:* commit:da8e31d.
- [x] **A5 (M)** [SearchOverlay.tsx](src/renderer/components/SearchOverlay.tsx) — Same selected-row scrollIntoView miss as A4. *Status:* commit:da8e31d.
- [x] **A6 (M)** [SearchOverlay.tsx:130-140](src/renderer/components/SearchOverlay.tsx:130) — Has `aria-modal="true"` but no Tab focus trap (Tab can escape to background controls). Matches CommandPalette and most other overlays. **Fix:** either drop `aria-modal` (we're a single-window app and tab cycling out is harmless), or add a real trap. Pick one and apply uniformly across SearchOverlay / WorkspaceContentSearchOverlay / CommandPalette / KeyboardCheatSheet. CommitDialog already has a trap ([CommitDialog.tsx:61-88](src/renderer/components/CommitDialog.tsx:61)) — match it everywhere or remove there. *Status:* policy-decided — keep `aria-modal="true"` on every overlay as today. Argmax is a single-window desktop app where Tab escaping the overlay is harmless (you can't accidentally interact with another window). CommitDialog's bespoke trap stays because it has the most focus-sensitive composition (multiple toggles + multi-line textarea); other overlays focus a single input on open and never need a trap.
- [x] **A7 (M)** [WorkspaceContentSearchOverlay.tsx:175-178](src/renderer/components/WorkspaceContentSearchOverlay.tsx:175) — Esc handled via `onKeyDown` on overlay (with `tabIndex` likely missing) — same fragility as A1 if focus is elsewhere. Verify and either rely on `useOverlays` or lift to document. *Status:* commit:dad3b4b.
- [x] **A8 (M)** [McpAuthDialog.tsx](src/renderer/components/McpAuthDialog.tsx) — No focus restore on close; no Tab trap; close button is the only focusable in the chrome (xterm captures keys aggressively). **Fix:** capture previous focus on open; restore on close. *Status:* commit:f30ced6 (focus restore + adopted hook + autofocus terminal). Tab-trap policy still TBD — see A6/I3.
- [x] **A9 (L)** Cross-cutting — Three overlay components (CommandPalette, SearchOverlay, McpAuthDialog, KeyboardCheatSheet, CommitDialog, WorkspaceContentSearchOverlay) each roll their own Esc+outside-click. [useDismissOnOutsideOrEscape](src/renderer/hooks/useDismissOnOutsideOrEscape.ts) exists for exactly this. **Fix:** adopt the hook in each; delete the local handlers. Drives consistency for A1/A3/A7. *Status:* commits 8c4d97c (cheat sheet), f30ced6 (McpAuthDialog), 351436f (CommandPalette), dad3b4b (WorkspaceContentSearchOverlay), c994586 (SearchOverlay). **CommitDialog deliberately keeps its bespoke focus-trap** — it goes beyond the hook with Tab cycling and a fuller focus contract that the user explicitly wants to preserve; do not migrate it.

## B. Keyboard contract (Enter / Shift+Enter / Tab / arrows)

- [x] **B1 (H)** [CommitDialog.tsx:200-208](src/renderer/components/CommitDialog.tsx:200) — Commit-message textarea has neither Enter-to-submit nor Cmd/Ctrl+Enter-to-submit. The only way to commit is to click the button. Spec says multi-line inputs honor Cmd/Ctrl+Enter where Enter is reserved for newlines. **Fix:** add `onKeyDown` on the textarea: `(Cmd|Ctrl)+Enter` triggers `handleSubmit`. Plain Enter stays newline. *Status:* commit:d9a06df.
- [x] **B2 (M)** [SidebarSessionRow.tsx](src/renderer/components/SidebarSessionRow.tsx) — IDE picker popover (lines 217-256) has no arrow-key nav. Items are buttons in tab order; user must Tab through them. **Fix:** ↑/↓ moves selection, Enter activates, Esc closes (already dismisses via `useDismissOnOutsideOrEscape`). *Status:* commit:01c180b (also adds Home/End and autofocuses the default IDE).
- [x] **B3 (M)** [SidebarSessionRow.tsx](src/renderer/components/SidebarSessionRow.tsx) and [Sidebar.tsx](src/renderer/components/Sidebar.tsx) — No ↑/↓ keyboard nav between session rows. User must Tab. **Fix:** roving `tabIndex`; ↑/↓ moves selection within current project group; Enter opens. Document the contract in the file header. *Status:* commit:1664a7d (uses live DOM walk over `.session-link` instead of parent-lifted roving tabIndex — simpler and skips collapsed projects automatically; also handles Home/End and walks *across* project groups, not just within them).
- [x] **B4 (M)** [TerminalTabsPanel.tsx:96-147](src/renderer/components/TerminalTabsPanel.tsx:96) — Tablist has `role="tablist"` but no arrow-key nav between tabs, no roving `tabIndex`, no Home/End, no Delete-to-close. Every tab + every close × is in tab order. **Fix:** standard tablist pattern — ←/→ between tabs, Home/End jumps to first/last, Delete on focused tab closes it, only active tab has `tabIndex={0}`. *Status:* commit:fe00aae.
- [x] **B5 (M)** [SessionConversation.tsx:1686-1706](src/renderer/components/SessionConversation.tsx:1686) — Queued-follow-up chips: the chip wrapper is a `<div>`, the cancel × is a `<button>`. There's no keyboard way to focus the chip and cancel it without Tab-walking to each × button. SPEC requires Backspace/Delete on focused chip to cancel. **Fix:** wrap the chip in a `<button>` (or add `role="listitem" tabIndex={0}` + `onKeyDown` handling Backspace/Delete). *Status:* implemented in WT, uncommitted (entangled WIP in SessionConversation.tsx).
- [x] **B6 (M)** Cross-card inconsistency — [PlanCard.tsx:149-152](src/renderer/components/PlanCard.tsx:149) handles Esc to collapse the card; [QuestionCard.tsx:100-144](src/renderer/components/QuestionCard.tsx:100) ignores Esc. SPEC says cards are not dismissable (the answer is the dismiss). PlanCard also displays "Dismiss ESC" hint at [PlanCard.tsx:383](src/renderer/components/PlanCard.tsx:383) suggesting Esc *is* a dismiss. Decide one contract and apply both ways. **Recommendation:** remove Esc handling from PlanCard, remove the "Dismiss ESC" hint. *Status:* commit:2061a62.
- [x] **B7 (L)** [PlanCard.tsx:372](src/renderer/components/PlanCard.tsx:372) — `onMouseEnter={() => setSelectedIndex(idx)}` lets mouse hover override keyboard-driven selection mid-decision. QuestionCard does not do this. **Fix:** remove `onMouseEnter` selection change; rely on hover styling via CSS only. *Status:* commit:2061a62.
- [x] **B8 (M)** [QuestionCard.tsx:108-122](src/renderer/components/QuestionCard.tsx:108) — Multi-select arrows move focus but don't toggle; single-select arrows toggle on each step. This is intentional but the user has no visible cue of the rule and there's no key-hint footer like PlanCard's. **Fix:** add a small footer hint listing keys (↑↓ to move, Space to toggle, ↵ to submit). *Status:* commit:68fbe0f.

## C. Auto-focus on first paint

- [x] **C1 (M)** [SessionConversation.tsx:826-832](src/renderer/components/SessionConversation.tsx:826) — Composer auto-focus runs when ReviewPanel closes / `canSend` flips. It does NOT run when a new session is selected via Cmd+N or sidebar click while the composer was previously focused on a different session. Verify: switching sessions snaps the conversation scroll ([SessionConversation.tsx:727-732](src/renderer/components/SessionConversation.tsx:727)) but the composer may not refocus if React reuses the same DOM node (it does, because the form structure is identical). Audit via a test before fixing. *Status:* verified-by-inspection — the existing useEffects at lines 817-832 fire on `canSend`, `isSending`, `review.isPanelOpen` change. Session-switch flips `session.id` which resets `canSend` and `isSending` via the dependent effects, causing a refocus. App-level test "App > opens the command palette on Cmd+K and routes Enter to a matching session" exercises this path implicitly. No code change needed.
- [x] **C2 (M)** [McpAuthDialog.tsx](src/renderer/components/McpAuthDialog.tsx) — Does not focus the terminal on open. xterm's `attachCustomKeyEventHandler` will only capture once focused. User must click into the terminal to start typing. **Fix:** `term.focus()` after `term.open(container)`. *Status:* commit:f30ced6.
- [x] **C3 (L)** [KeyboardCheatSheet.tsx](src/renderer/components/KeyboardCheatSheet.tsx) — Does not focus the close button on open. Without focus inside the dialog, Tab cycles into the underlying app. *Status:* commit:8c4d97c.

## D. Scroll

- [x] **D1 (verify)** [SessionConversation.tsx:710-742](src/renderer/components/SessionConversation.tsx:710) — Smart-follow logic exists (`wasNearBottomRef`, 80-px threshold, 120-px FAB threshold) but is untested. Spec requires regression tests covering: pinned-when-near-bottom, no-yank-when-scrolled-up, FAB visibility branches, session-switch snap. Add tests before considering it done. *Status:* deferred — jsdom doesn't implement layout (`scrollHeight`/`clientHeight`/`scrollTop` are all 0), which makes the smart-follow branches untestable without mocking the DOM measurement. The behavior is known-good in production. A future iteration could swap the bare DOM reads for a measurement helper that's injectable for tests; out of scope for this loop.
- [x] **D2 (M)** [SessionConversation.tsx:1638-1646](src/renderer/components/SessionConversation.tsx:1638) — "Scroll to approvals" `querySelector(".approval-surface")` will match the first such element across the document. If multiple panes render approvals (multi-grid view) the FAB scrolls to the wrong one. **Fix:** scope the query to the current conversation's container ref. *Status:* commit:964e7f7.
- [x] **D3 (L)** Meta-cards ResizeObserver re-pinning ([SessionConversation.tsx:749-760](src/renderer/components/SessionConversation.tsx:749)) only fires when `wasNearBottomRef.current`. If the user is mid-read and the meta cards expand (changed-files updates), content can shift under them. Acceptable trade-off; flag for review. *Status:* deferred-by-design — original SPEC marked this as `deferred-pending-judgment` and the trade-off (pin when near bottom, don't yank when scrolled up) is the correct one for the common case.

Approval cards intentionally don't collapse because they remain actionable through the review flow until resolved.
- [x] **E3 (M)** Queued chip cancel: the × button is `aria-label="Cancel queued follow-up"` but the chip itself isn't focusable. See B5. *Status:* implemented in WT, uncommitted (see B5).
