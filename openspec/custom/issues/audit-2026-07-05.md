Overall: the codebase is in good shape. The largest risk is a real provider lifecycle race in the follow-up launch path. Most other issues are stale migration plumbing, dead abstractions, or places where the UI/docs promise more than the Rust backend now provides.

## Findings

### H1. Follow-up relaunch can survive Stop during spawn

- File: `src-tauri/src/providers/session_service.rs:553`
- Pattern: race condition / resource leak
- Problem: `send_input()` handles a completed session by inserting `HandleEntry::Pending`, awaiting `launcher.launch()`, and then unconditionally inserting `HandleEntry::Resolved` at `src-tauri/src/providers/session_service.rs:579`. If `terminate()` runs while the launch future is in flight, it removes the pending entry and records the session as cancelled, but this path then stores the freshly spawned handle anyway. The initial `launch()` path already has the correct guard at `src-tauri/src/providers/session_service.rs:297`.
- Impact: a provider process can keep running after the user hit Stop on a follow-up/relaunch. On launch error, the same path can also overwrite a cancellation with a launch failure.
- Recommendation: copy the initial launch guard into `send_input()`: after `launcher.launch()` resolves, install the handle only if the previous entry was still `Pending`; otherwise remove the resolved entry, call `handle.terminate().await`, and return. On launch error, call `record_launch_failure()` only when the removed entry was still `Pending`. Add a regression test for "terminate during follow-up spawn".

### M1. Invalid event payload JSON is silently replaced with `{}`

- File: `src-tauri/src/persistence/events.rs:273`
- Pattern: silent failure / data quality
- Problem: `event_row_to_timeline_event()` parses persisted `payload_json` with `unwrap_or_else(|_| serde_json::json!({}))`. A corrupt or partially written payload becomes an empty object without any error, log, or visible marker.
- Impact: the timeline can lose tool metadata, approval details, usage payloads, or provider-specific fields while still looking like a valid event. That makes data corruption harder to notice and harder to debug.
- Recommendation: make this loud. Return a row conversion error, or preserve a sentinel payload such as `{ "parseError": true, "rawPayload": "...truncated..." }` and log the row id/session id. Add a test that inserts malformed `payload_json` and confirms the read path exposes the problem.

### M2. Diagnostics log export is dead plumbing

- Files: `src-tauri/src/ipc/system.rs:164`, `src/renderer/components/settings/SystemSettings.tsx:90`, `src/renderer/components/settings/SystemSettings.tsx:168`, `src/shared/logger.ts:13`
- Pattern: stale docs / broken diagnostics UI
- Problem: `system:diagnostics` always returns `recent_logs: Vec::new()`, while the Settings UI has a "Save log file" button and a "Recent logs" section based on `diagnostics.recentLogs`. The shared logger comment still says the diagnostics panel reads the main-side buffer through `system:diagnostics`, but the main process is now Rust and does not use that TypeScript logger.
- Impact: bug reports copied from Settings omit logs, and the "Save log file" action is always disabled for system diagnostics.
- Recommendation: either wire Rust tracing/log buffering into `DiagnosticsReport.recent_logs`, or remove `recentLogs`, the save-log button, and the stale `logger.ts` main-process wording. If logs stay, add a backend test that emits a log and sees it in diagnostics.

### L1. `Dispatcher` / `EventNormalizer` is dead production abstraction

- Files: `src-tauri/src/providers/normalizer/mod.rs:150`, `src-tauri/src/providers/flush_queue.rs:167`, `src-tauri/src/providers/flush_queue.rs:238`
- Pattern: dead code / duplicate buffering logic
- Problem: `EventNormalizer` and `Dispatcher` maintain their own per-session line buffers, but production normalization goes through `ProviderEventFlushQueue`, which calls `normalize_provider_event()` directly. `Dispatcher` is now used by tests and test imports, not the runtime path.
- Impact: there are two buffering implementations to reason about. Future fixes can land in the unused dispatcher path and miss production, or vice versa.
- Recommendation: delete `EventNormalizer`/`Dispatcher`, or move them under `#[cfg(test)]`. Prefer tests that exercise `ProviderEventFlushQueue`, since that is the runtime owner of buffering.

### L2. Leading-dash file paths are rejected even after `--`

- Files: `src-tauri/src/ipc/validation.rs:426`, `src-tauri/src/git/exec.rs:332`
- Pattern: over-strict validation / edge-case UX bug
- Problem: `validate_relative_path()` rejects any path starting with `-`, and `validate_pathspec_args()` rejects pathspecs after the explicit `--` separator. This blocks valid Git paths such as `-notes.md`, even though `--` is the standard way to pass them safely.
- Impact: rare but valid repo files cannot be opened, searched, reviewed, or committed through Argmax.
- Recommendation: allow leading dashes for relative file paths once every Git pathspec call uses `--`. Keep leading-dash rejection for refs and option-like user arguments. Add tests for reading/reviewing/committing a file named `-notes.md`.

### L3. Hand-written IPC return types have already drifted

- Files: `src/shared/types.ts:441`, `src/renderer/lib/tauriBridge.ts:135`, `src-tauri/src/ipc/projects.rs:60`
- Pattern: generated type drift / stale compatibility layer
- Problem: the hand-written `ArgmaxApi.projects.remove()` type says it returns `{ projectId: string }`, and `tauriBridge.ts` invokes it with that shape. The Rust command returns `ArgmaxResult<()>`, so the real value is unit/null. Current callers ignore the value, which is why this slipped through.
- Impact: this is harmless today, but it proves the hand-written bridge can lie even when channel parity and generated binding freshness checks pass.
- Recommendation: change `projects.remove` to `Promise<void>` or `Promise<null>` in the handwritten API, and then tighten the bridge around generated command result types so this class of drift is harder to reintroduce.

## Cleanup Recommendations

1. Fix H1 first. It is the only finding that can leak live provider work after a user cancellation.
2. Make malformed persisted JSON visible before adding more timeline features.
3. Decide whether Settings diagnostics should include backend logs. If yes, build the Rust log buffer; if no, remove the UI/docs promise.
4. Delete or test-gate old migration scaffolding: `Dispatcher`, audit-stamped comments, and "matches TS service" references that no longer explain live architecture.
5. Relax leading-dash path handling with focused safety tests.
6. Use generated IPC return types for the renderer bridge where practical. The current parity check catches missing channels, not wrong return shapes.
